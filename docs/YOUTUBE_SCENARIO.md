## 📹 4. 데이터 파이프라인 - 배치 시스템 (5:00 - 8:00)

### [화면 - 슬라이드]
- 제목: "배치 파이프라인 (8단계)"
- 배경: 진행 막대 모양의 시각화

### [자막/그래픽]
```
ODIN-AI 배치 시스템: 8단계 파이프라인

Phase 1: 데이터 수집
├─ 나라장터 공공데이터포털 API 호출
├─ 날짜 범위: 최근 7일
├─ 수집량: 하루 평균 50-100개 공고
└─ 저장소: bid_announcements 테이블

Phase 2: 문서 다운로드
├─ HWP/PDF/HWPX 파일 다운로드
├─ 로컬 스토리지에 저장
├─ 성공률: 95%+ (일부 파일 포맷 문제)
└─ 저장 경로: storage/documents/

Phase 3: 문서 처리
├─ HWP → 마크다운 변환
├─ 테이블/리스트 파싱
├─ 정보 추출 (가격, 일정, 자격요건 등)
├─ 태그 자동 생성 (건설/IT/물품/조경 등)
└─ 결과 저장: bid_extracted_info 테이블

Phase 3.5: 벡터 임베딩 생성
├─ 문서를 1024 토큰 단위로 청킹
├─ OpenAI text-embedding-3-small 모델 사용 (1536차원)
├─ 또는 KURE-v1 로컬 모델 (1024차원)
├─ pgvector HNSW 인덱스로 저장
└─ 결과 저장: rfp_chunks 테이블

Phase 3.6: Neo4j 그래프 동기화
├─ bid_announcements → BidAnnouncement 노드 생성
├─ Organization 노드 생성
├─ BID_ISSUED_BY 관계 생성
├─ SIMILAR_TO 엣지 생성 (코사인 유사도 > 0.7)
└─ 총 노드: 23,000+ / 엣지: 16,000,000+

Phase 3.7: GraphRAG 엔티티 추출
├─ Ollama 로컬 LLM (EXAONE 3.5:7.8b) 사용
├─ RFP 문서에서 엔티티 추출 (조직, 기술 용어, 프로젝트명)
├─ 엔티티 간 관계 학습
└─ 커뮤니티 감지 알고리즘 적용

Phase 4: 알림 매칭
├─ 새 입찰 vs 사용자 알림 규칙 매칭
├─ 조건: 키워드, 가격 범위, 지역, 업종
├─ 사용자별 알림 집계
└─ SMTP로 이메일 발송

Phase 5: 보고서 생성
├─ 배치 실행 결과를 JSON으로 저장
├─ 통계 요약 (수집/처리/성공 건수)
├─ 관리자에게 이메일 리포트 발송
└─ reports/ 디렉토리에 저장
```

### [내레이션]
"이제 ODIN-AI의 심장인 배치 시스템을 자세히 설명하겠습니다. 매일 밤 실행되는 8개의 단계로 이루어져 있습니다.

**Phase 1: 데이터 수집**

먼저 공공데이터포털 API를 호출합니다. 최근 7일 동안 올라온 모든 입찰공고를 가져오죠. 하루에 평균 50~100개 정도입니다. 이들은 bid_announcements 테이블에 저장됩니다.

**Phase 2: 문서 다운로드**

각 공고는 RFP 문서를 가지고 있습니다. HWP, PDF, HWPX 형식의 파일들이죠. 배치 시스템이 이들을 모두 다운로드합니다. 성공률은 95% 이상입니다.

**Phase 3: 문서 처리**

다운로드한 파일들을 마크다운 형식으로 변환합니다. 테이블과 리스트를 파싱하고, 중요 정보를 추출합니다. 예정가격, 공사 기간, 자격 요건, 특수 조건 등이죠. 동시에 태그를 자동으로 생성합니다.

**Phase 3.5: 벡터 임베딩 생성**

이제 AI의 역할이 시작됩니다. 처리된 문서들을 약 1024개 토큰 단위로 잘라냅니다. 각 청크를 OpenAI의 text-embedding-3-small 모델로 벡터화합니다. 또는 로컬 KURE-v1 모델을 사용할 수도 있죠. 벡터는 PostgreSQL의 pgvector 확장에 HNSW 인덱스로 저장됩니다. 이렇게 하면 나중에 매우 빠른 유사도 검색이 가능합니다.

**Phase 3.6: Neo4j 그래프 동기화**

동시에 그래프 데이터베이스도 업데이트됩니다. 각 입찰공고는 BidAnnouncement 노드가 되고, 발주 기관은 Organization 노드가 됩니다. 노드들 사이에는 여러 종류의 관계(엣지)가 생성되는데, 특히 중요한 것은 SIMILAR_TO 엣지입니다. 코사인 유사도가 0.7 이상인 공고들끼리 연결되죠. 현재 우리는 23,000개 이상의 노드와 1,600만 개 이상의 엣지를 가지고 있습니다.

**Phase 3.7: GraphRAG 엔티티 추출**

좀 더 고급진 작업도 합니다. Ollama 로컬 LLM을 사용해서 RFP 문서에서 중요한 엔티티들을 추출합니다. 조직명, 기술 용어, 프로젝트명 등이죠. 엔티티 간의 관계를 학습하고, 커뮤니티 감지 알고리즘을 적용합니다. 이렇게 하면 나중에 그래프 기반 검색이 더욱 정교해집니다.

**Phase 4: 알림 매칭**

이제 핵심적인 비즈니스 로직입니다. 새로 수집한 입찰들을 모든 사용자의 알림 규칙과 비교합니다. 규칙에는 여러 조건이 있을 수 있습니다. 특정 키워드를 포함해야 하고, 가격이 일정 범위 내에 있어야 하고, 특정 지역이어야 하며, 특정 업종이어야 하는 식이죠. 조건에 맞는 입찰들을 찾아서 notification 테이블에 기록합니다.

**Phase 5: 보고서 생성**

마지막으로 배치 실행 결과를 정리합니다. JSON 형식의 보고서를 생성해서 reports 디렉토리에 저장하고, 관리자에게 이메일로 발송합니다."

## 📹 5. AI 검색 시스템 (8:00 - 11:00)

### [화면 - 슬라이드 1]
- 제목: "3가지 검색 패러다임"
- 배경: 3개의 다른 색상 영역

### [자막/그래픽]
```
ODIN-AI 검색 시스템

1️⃣ 키워드 검색
   └─ PostgreSQL LIKE + FTS + pg_trgm

2️⃣ RAG 하이브리드 검색
   └─ 벡터 유사도 + 전문 검색 + RRF 통합

3️⃣ 그래프 검색
   └─ Neo4j Cypher + 관계 기반 발견
```

### [내레이션]
"ODIN-AI는 3가지 검색 방식을 제공합니다. 각각 다른 상황에 최적화되어 있습니다."

### [화면 - 슬라이드 2]
- 제목: "검색 패러다임 1: 키워드 검색"
- 배경: 파란색

### [자막/그래픽]
```
🔍 키워드 검색 (Traditional)

사용자 입력: "도로 포장 공사"

Step 1: PostgreSQL LIKE
  └─ title LIKE '%도로%' OR title LIKE '%포장%'

Step 2: Full Text Search (FTS)
  └─ SELECT * FROM bid_announcements
     WHERE title_tsv @@ plainto_tsquery('korean', '도로 포장')

Step 3: 유사도 계산
  └─ pg_trgm 확장으로 문자열 유사도 계산
  └─ SELECT similarity(title, '도로 포장 공사')

결과: 정확한 일치와 부분 일치 조합

✓ 장점: 빠름, 정확함, 다국어 지원
✗ 단점: 의미 있는 변형어 미감지
```

**첫째, Graceful Degradation(우아한 실패)**. 한 서비스가 실패해도 전체 시스템이 동작합니다. 예를 들어, OpenAI API가 다운되어 임베딩 생성이 실패해도, 키워드 검색은 계속 작동합니다. 사용자는 약간의 기능 저하만 경험하지, 완전한 장애는 경험하지 않습니다.

**둘째, Strategy Pattern(전략 패턴)**. 임베딩 생성에 여러 가지 방식이 있을 수 있습니다. OpenAI를 사용할 수도 있고, KURE를 사용할 수도 있습니다. 이들을 모두 같은 인터페이스로 구현해서, 환경변수로 쉽게 전환할 수 있게 했습니다.

**셋째, Pipeline/Orchestrator(파이프라인)**. 배치 프로세스의 8개 단계는 모두 독립적입니다. 각 단계는 환경변수로 활성화/비활성화할 수 있습니다. 그리고 Advisory Lock을 사용해서 동시에 배치가 실행되지 않도록 보호합니다.

**넷째, Lazy Loading(지연 로딩)**. 프론트엔드의 각 페이지는 따로따로 로딩됩니다. 첫 페이지만 빠르게 로드되고, 다른 페이지는 사용자가 접근할 때 로드됩니다.

**다섯째, RRF(Reciprocal Rank Fusion)**. 여러 검색 방식의 결과를 지능적으로 합칩니다. 벡터 검색에서 1위가 나왔지만 텍스트 검색에서는 10위였다면, RRF는 가중 평균을 계산해서 최종 순위를 정합니다.

**여섯째, Advisory Lock**. PostgreSQL의 권고 잠금 기능을 사용합니다. 배치 프로세스가 실행 중일 때, 다른 배치가 동시에 실행되지 않도록 합니다."
